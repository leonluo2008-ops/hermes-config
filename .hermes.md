# hermes-config

> 一句话：Hermes Agent 配置管理套件——2 个 skill + 1 个 plugin，规范 SOUL.md / .hermes.md / MEMORY.md 三层配置架构，并自动执行配置自检、信息路由建议、会话复盘。

## 关键约束（开发铁律）

- **通用性优先，拒绝本机硬编码**——仓库里的代码 / 文档 / 命令示例必须可迁移到任意一台新机器。
  - ✅ 用 `$HOME` / `~` / 相对路径 / 环境变量（`HERMES_HOME` / `HERMES_SKILLS_DIR` / `HERMES_PLUGINS_DIR`）
  - ❌ 禁止 `/home/luo`、`/Users/xxx`、`~/Github/`（`~/Github/` 也是硬编码——别的机器可能在 `~/code/` 或 `~/projects/`）
  - 命令示例：用 `./install.sh`、`<repo-dir>/install.sh`，不要写 `~/Github/hermes-config/install.sh`
  - 已知存量：仓库内目前有 ~17 处本机绝对路径（GUIDE.md / local-to-github-sync.md / SKILL.md / README.md），需要逐步清理
- **改 skill 内容前必须读官方文档核实**——`hermes-config-organization` 和 `hermes-md-init` 讲的是 Hermes 配置文件的真实行为（截断上限、搜索规则、优先级），凭印象写 = 违规。官方文档：
  - Context Files: https://hermes-agent.nousresearch.com/docs/context-files
  - Personality & SOUL.md: https://hermes-agent.nousresearch.com/docs/personality
  - Memory: https://hermes-agent.nousresearch.com/docs/memory
  - Plugins: https://hermes-agent.nousresearch.com/docs/plugins
- **config-advisor 是 observer-only，永不阻断**——与 harness-guard（阻断式审查）互补共存。改 hook 时不要破坏这个边界（不用 `transform_tool_result`，那是 harness-guard 的 first-wins 领地）。

## 版本维护规则

每次发布（push 到 main）前必须三步走：

1. **打 tag**（语义化版本）：
   - skill 大改（新增章节 / 重写流程）→ minor bump（v3 → v4）
   - 文档修正 / typo / 小补充 → patch bump（v3.0 → v3.1）
   - plugin 行为变化（hook 逻辑 / 阈值 / 新模块）→ 同步 bump plugin.yaml 的 version 字段
   ```bash
   git tag -a v3.1 -m "v3.1: 修复 config-health-check.md 字节口径"
   git push origin v3.1
   ```
2. **更新 CHANGELOG.md**（Keep a Changelog 格式，放仓库根目录）
3. **校对 README.md**——如果改了安装流程 / 依赖 / 配置项，README 必须同步

## 登记清单（代码变更同步更新）

改完代码后，根据改动范围同步以下位置。**清单本身也要随代码变化调整**——新增模块时加新条目。

| 改了什么 | 必须同步登记的位置 |
|---------|------------------|
| skill 的 description / 触发词 | `hermes-config-organization/SKILL.md` 或 `hermes-md-init/SKILL.md` 的 frontmatter |
| config-advisor 的 hook 逻辑 | `config-advisor/docs/GUIDE.md`（保姆级文档）+ `config-advisor/README.md` |
| config-advisor 的健康检查阈值 | `hermes-config-organization/references/config-health-check.md`（skill 与 plugin 必须一致）|
| 新增 / 删除文件 | `install.sh`（如果是可安装产物）+ README 目录结构段 |
| 安装流程变更 | `README.md` 安装段 + `config-advisor/docs/GUIDE.md` §5 |
| 新增 Python 依赖 | `config-advisor/requirements.txt`（待创建）+ README 前置条件 + install.sh 自动安装 |

## 非显然命令

- **安装 skills**：`./install.sh`（只装 skill）
- **安装 skills + plugin**：`./install.sh --plugins`（显式 opt-in，plugin 需要先装好 skill）
- **同步开发改动到运行目录**（开发循环）：`cp -r config-advisor/ $HERMES_PLUGINS_DIR/` 或重跑 `./install.sh --plugins`
- **外部终端重启 gateway**：`hermes gateway restart`（不能从 agent 内部重启，会自杀）
- **验证两边一致**：`diff config-advisor/health_check.py $HERMES_PLUGINS_DIR/config-advisor/health_check.py`

## 环境约定

- **仓库不是运行目录**——`install.sh` 用 `cp -r` 复制到运行目录（`$HERMES_HOME/skills/` + `$HERMES_HOME/plugins/`），不是软链。两边是独立副本。改代码永远在仓库，运行目录的改动会被下次 install 覆盖。
- **新机器 clone 后的初始化**：
  1. `git clone <repo>` 到任意目录
  2. `cd <repo> && ./install.sh --plugins`
  3. `cp config-advisor/.env.example config-advisor/.env`，填入 `CONFIG_ADVISOR_API_KEY`（或 fallback：`ZAI_API_KEY` / `GLM_API_KEY` 等）
  4. Python 依赖 `httpx`（复盘报告 LLM 调用用）—— **Hermes 自带 venv 已包含**（`~/.hermes/hermes-agent/venv/`），无需单独安装。仅独立测试时按 `config-advisor/requirements.txt` 装
  5. 在 `~/.hermes/config.yaml` 的 `plugins.enabled` 列表加 `config-advisor`
  6. 外部终端 `hermes gateway restart`
- **多机器同步**：按用户需求灵活处理，默认靠手动 `git pull`，没有自动化。

### 开发循环（三段式）

```
仓库改代码 → commit + push → install.sh 复制到运行目录 → 外部重启 gateway

```

## 配套全局规则继承（Step 3.5 必做）

> ⚠️ 项目级 `.hermes.md` 建立后，`~/.hermes.md` 在本目录下**完全被遮蔽**。以下全局关键规则在本项目继续生效。

### 关键原则（搬运自 `~/.hermes.md`）

- **主动推送，不要等待追问**——定时任务完成后主动推送结果
- **系统问题优先处理**——Gateway 崩了要先恢复，再做其他事
- **效率优先**——能自动化的不手动，能并行的不串行

### Hindsight 记忆协议（摘要）

完成基础设施变更（Docker / systemd / cron / 反代 / 端口 / 数据库 schema）后，retain 一条 `infra_state` 到 Hindsight。具体调用格式见 `~/.hermes.md` 的"Hindsight 协议"段（或 `hindsight-helper-usage` skill）。本项目常见触发：改 config-advisor 阈值 / 新增 hook / 改 install.sh 安装路径。

### 凭印象病黑名单（摘要）

写任何具体值前必先 query 验证（grep / cat / read_file / `hermes config show`），写完回读验证。详见 `~/.hermes.md` 铁律 6 + 7。本项目高频踩坑：
- 凭印象写 Hermes 官方行为（截断上限、搜索规则）→ 必须读官方文档
- 凭印象写 Python / bash 语法 → 先 `--help` 或查文档
- 凭印象写绝对路径 → 用 `pwd` / `git rev-parse` 查实际路径

### 工作流程路由（本项目修订）

**本项目不涉及跨 agent 路由**——纯 Hermes 配置开发。全局 `~/.hermes.md` 里"内容创作→转发 huiben"的说法是**错误的**：不同 profile / 不同 gateway 之间不存在任务转交关系，各自独立运行。huiben 是独立 profile，不是当前 agent 的下属。

## 不写什么（agent 自己能拿到）

- 目录树 / 文件列表（agent 会自己 `ls` / `find`）
- Python 代码风格（交给 linter）
- skill / plugin 的 frontmatter 字段（从 `SKILL.md` / `plugin.yaml` 直接读）
- 临时 TODO 状态、commit SHA、PR 号（用 `git log` / `session_search` 回忆）
